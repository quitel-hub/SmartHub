## 1. Твірні патерни (Creational Patterns)

### 1.1 Singleton 
* **Місце використання:** `core/thread_pool.py` (клас `ProcessorPool`).
* **Призначення:** Створення єдиної точки доступу до пулу потоків для паралельної обробки OCR-запитів.
* **Обґрунтування:** Обробка зображень — ресурсомісткий процес. Наявність кількох пулів призвела б до неконтрольованого споживання RAM та CPU.
* **Переваги:** Економія ресурсів, глобальний доступ до черги завдань.
* **Недоліки:** Труднощі при написанні юніт-тестів через глобальний стан.

### 1.2 Factory Method 
* **Місце використання:** `core/document_processor.py` (клас `ProcessorFactory`).
* **Призначення:** Створення об'єктів-процесорів залежно від типу документа (`math_exam`, `standard_text`).
* **Обґрунтування:** Клієнтський код (бот) не повинен знати деталі ініціалізації конкретних обробників.
* **Переваги:** Дотримання принципу Open/Closed — нові типи документів додаються без зміни фабрики.
* **Недоліки:** Ускладнення коду при невеликій кількості типів об'єктів.

### 1.3 Builder 
* **Місце використання:** `core/report_builder.py` (клас `ReportBuilder`).
* **Призначення:** Покрокове формування складного об'єкта `OCRReport`.
* **Обґрунтування:** Звіт складається з багатьох частин (Header, Content, Metadata, Footer), які можуть додаватися в різному порядку.
* **Переваги:** Чіткий розподіл процесу конструювання та репрезентації об'єкта.
* **Недоліки:** Необхідність створення додаткового класу для кожного виду продукту.

## 2. Структурні патерни (Structural Patterns)

### 2.1 Facade
* **Місце використання:** `core/ocr_engine.py` (клас `OCREngine`).
* **Призначення:** Надання простого інтерфейсу для роботи з бібліотеками Tesseract та OpenCV.
* **Обґрунтування:** Бібліотеки OCR мають складне API з багатьма параметрами. Фасад ховає цю складність за одним методом `extract_text()`.
* **Переваги:** Зменшення залежностей клієнтського коду від зовнішніх бібліотек.
* **Недоліки:** Ризик перетворення фасаду на "божественний об'єкт" (God Object).

### 2.2 Adapter 
* **Місце використання:** `core/google_sheets_adapter.py` (клас `GoogleSheetsAdapter`).
* **Призначення:** Адаптація об'єкта `OCRReport` до формату рядків, який вимагає Google Sheets API.
* **Обґрунтування:** Зовнішні сервіси мають власні формати даних, які не збігаються з внутрішньою логікою програми.
* **Переваги:** Можливість інтегрувати будь-який сторонній сервіс без зміни ядра системи.
* **Недоліки:** Додаткові витрати пам'яті на конвертацію об'єктів.

### 2.3 Decorator 
* **Місце використання:** `core/text_decorators.py` (класи `StripWhitespaceDecorator`, `AutoCorrectDecorator`).
* **Призначення:** Динамічне додавання функцій постобробки тексту (видалення пробілів, корекція типових помилок OCR).
* **Обґрунтування:** Користувач може сам обирати, які фільтри застосувати до розпізнаного тексту.
* **Переваги:** Гнучкість у комбінуванні поведінки.
* **Недоліки:** Складність у відстеженні порядку викликів декораторів.

### 2.4 Composite
* **Місце використання:** `core/composite.py` (класи `SinglePageDocument`, `MultiPageDocument`).
* **Призначення:** Обробка окремих сторінок та багатосторінкових конспектів за єдиним інтерфейсом.
* **Обґрунтування:** Система має однаково сприймати одну фотографію та папку з фотографіями.
* **Переваги:** Спрощення клієнтського коду через рекурсивну обробку структур.
* **Недоліки:** Труднощі в обмеженні типів елементів, що входять до композиції.

## 3. Поведінкові патерни (Behavioral Patterns)

### 3.1 Template Method
* **Місце використання:** `core/document_processor.py` (абстрактний метод `process_document`).
* **Призначення:** Визначення скелета алгоритму обробки (Load -> Preprocess -> OCR -> Postprocess).
* **Обґрунтування:** Кроки обробки стабільні, але їх реалізація (Preprocess) залежить від типу документа.
* **Переваги:** Уникнення дублювання коду в спадкоємцях.
* **Недоліки:** Жорстка прив'язка до структури, визначеної в базовому класі.

### 3.2 Strategy 
* **Місце використання:** `core/document_processor.py` (інтерфейс `OCRStrategy` та класи `StandardTextStrategy`, `MathExamStrategy`).
* **Призначення:** Зміна алгоритму розпізнавання (текст vs формули) під час виконання.
* **Обґрунтування:** Різні типи документів потребують різних налаштувань двигуна розпізнавання.
* **Переваги:** Чиста ізоляція алгоритмів один від одного.
* **Недоліки:** Клієнт повинен знати про існування різних стратегій для їх вибору.

### 3.3 State
* **Місце використання:** `bot/states.py` (клас `UserState`) та `bot/handlers.py`.
* **Призначення:** Керування поведінкою бота залежно від етапу взаємодії (Idle, Processing).
* **Обґрунтування:** Запобігання повторним запитам під час обробки важкого файлу (анти-спам).
* **Переваги:** Логіка переходів зосереджена в одному місці.
* **Недоліки:** Складність при великій кількості станів.

### 3.4 Command 
* **Місце використання:** `core/commands.py` (класи `StartCommand`, `HelpCommand`).
* **Призначення:** Перетворення команд користувача (/start, /help) на об'єкти.
* **Обґрунтування:** Поділ ініціатора команди та її безпосереднього виконавця.
* **Переваги:** Можливість чергування, логування та скасування дій.
* **Недоліки:** Велика кількість дрібних класів-команд.

### 3.5 Observer 
* **Місце використання:** `core/observer.py` (клас `DocumentEventManager`) та `bot/handlers.py`.
* **Призначення:** Сповіщення підсистем (Telegram UI, Google Sheets) про завершення обробки документа.
* **Обґрунтування:** Обробник документа не повинен знати, куди саме відправляються результати.
* **Переваги:** Низька зв'язність (Low Coupling).
* **Недоліки:** Складність у відстеженні ланцюжка подій в асинхронному середовищі.
